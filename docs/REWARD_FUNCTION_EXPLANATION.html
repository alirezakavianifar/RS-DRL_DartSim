DARTSim Reward Function

Overview

The reward function is a multi-objective weighted reward that combines:

  • Mission success
  • Targets detected
  • Team survival
  • Decision efficiency

It uses sparse rewards: step penalty during episode, terminal reward at end.

Implementation

Code Location

dartsim_env.py, method _compute_reward()

Reward Structure

def _compute_reward(self, results: Dict[str, Any], done: bool) -> float:
    """
    Compute reward based on Phase 0 design.
    
    Multi-objective weighted reward:
    - Mission success: 0.4
    - Targets detected: 0.3
    - Team survival: 0.2
    - Decision efficiency: 0.1
    """
    if not done:
        # Sparse step penalty
        return -0.01
    
    # Terminal reward
    reward = 0.0
    w = self.reward_weights
    
    # Mission success bonus
    if results.get("missionSuccess", False):
        reward += w["w_success"] * 1.0
    
    # Targets detected (normalized)
    targets = results.get("targetsDetected", 0)
    targets_norm = min(targets / self.max_targets, 1.0)
    reward += w["w_targets"] * targets_norm
    
    # Team survival
    destroyed = results.get("destroyed", False)
    if destroyed:
        reward += w["w_destruction"] * (-1.0)
    else:
        reward += w["w_survival"] * 1.0
    
    # Decision time efficiency (lower is better)
    decision_time = results.get("decisionTimeAvg", self.max_decision_time)
    decision_time_norm = 1.0 - min(decision_time / self.max_decision_time, 1.0)
    reward += w["w_efficiency"] * decision_time_norm
    
    # Normalize to [-1, 1]
    return np.clip(reward, -1.0, 1.0)

Reward Components

Default Weights

reward_weights = {
    "w_success": 0.4,      # Mission success (highest priority)
    "w_targets": 0.3,      # Targets detected
    "w_survival": 0.2,     # Team survival bonus
    "w_efficiency": 0.1,    # Decision time efficiency
    "w_destruction": 0.5   # Penalty for team destruction
}

1. Step Reward (During Episode)

When: done = False

reward = -0.01
  • Purpose: Encourage finishing quickly
  • Value: -0.01 per step
  • Range: Negative (penalty)

Example:

  • 500 steps = 500 × (-0.01) = -5.0

2. Terminal Reward (Episode End)

When: done = True (mission complete or destroyed)

Component 1: Mission Success

if mission_success:
    reward += 0.4 * 1.0 = +0.4
  • Weight: 0.4 (highest)
  • Value: +0.4 if mission succeeds, 0 otherwise

Component 2: Targets Detected

targets_norm = min(targets_detected / max_targets, 1.0)
reward += 0.3 * targets_norm
  • Weight: 0.3
  • Normalization: targets / max_targets (capped at 1.0)
  • Range: 0.0 to 0.3
  • Example:
    • 3 targets detected out of 10 max → 0.3 × (3/10) = +0.09
    • 10 targets detected → 0.3 × 1.0 = +0.3

Component 3: Team Survival

if destroyed:
    reward += 0.5 * (-1.0) = -0.5  # Penalty
else:
    reward += 0.2 * 1.0 = +0.2     # Bonus
  • Survival bonus: +0.2
  • Destruction penalty: -0.5
  • Priority: Survival is important (destruction penalty > survival bonus)

Component 4: Decision Efficiency

decision_time_norm = 1.0 - min(decision_time / max_decision_time, 1.0)
reward += 0.1 * decision_time_norm
  • Weight: 0.1 (lowest)
  • Normalization:
    • Fast decisions (low time) → high efficiency → high reward
    • Slow decisions (high time) → low efficiency → low reward
  • Range: 0.0 to 0.1
  • Example:
    • Decision time = 100ms, max = 1000ms → 1.0 - 0.1 = 0.9 → +0.09
    • Decision time = 1000ms → 1.0 - 1.0 = 0.0 → +0.0

Final Clipping

reward = np.clip(reward, -1.0, 1.0)

All rewards are clamped to [-1.0, 1.0] range.

Reward Characteristics

Sparse Rewards

  • Most signal comes at episode end
  • Step penalty is minimal (-0.01)
  • Makes learning challenging (reward is delayed)

Multi-Objective

  • Balances multiple goals
  • Weights can be tuned
  • Mission success prioritized (0.4 weight)

Normalized Range

  • All rewards clipped to [-1, 1]
  • Prevents reward explosion
  • Stable for DQN learning

Tuning Rewards

To improve learning, you could:

  1. Increase step penalty: Make -0.01 → -0.02 to encourage faster completion
  2. Add intermediate rewards: Reward progress (e.g., per target detected)
  3. Adjust weights: Increase w_success if mission completion is critical
  4. Reward shaping: Add small positive rewards for good states

However, current design matches RS-DRL paper methodology (sparse, multi-objective).
